# 关于 shadcn/ui

shadcn/ui 并非传统意义上的 UI 组件库，它代表了一种全新的 Web UI 组件分发理念。这是一套经过精雕细琢、遵循无障碍标准且高度可定制的 UI 组件模板集合。


![hero](https://github.com/shadcn-ui/ui/blob/main/apps/v4/public/opengraph-image.png)

## 核心定义：不是库，是拥有的代码

在大多数前端开发体验中，组件是通过 `npm` 或 `yarn` 引入的闭源依赖。这种依赖模式虽然便捷，但往往伴随着样式的难以覆盖、包体积的冗余以及对底层逻辑掌控力的缺失。

**shadcn/ui 彻底打破了这种模式。**

它将组件的原始源代码直接交付到你的手中。这不是一个你“安装”的项目，而是一个你“集成”并“拥有”的项目。你只需挑选所需的部分，通过复制粘贴或官方 CLI 工具将其注入到你本地的 `components/ui` 路径下。一旦代码进入你的项目，它就变成了你代码库的一部分，你可以像修改自己写的任何组件一样，去修改它的逻辑、HTML 结构或 Tailwind 样式。

## 设计理念与技术灵魂

### 1. 交互原语 (Primitives)

shadcn/ui 的底层交互逻辑由 **Radix UI** 驱动。Radix 提供了一套无头（Headless）组件，专注于解决最难处理的交互部分：键盘导航、焦点管理、Aria 属性以及跨浏览器的一致性。这意味着在 shadcn/ui 中，每一个看似简单的下拉菜单或对话框，在逻辑层面上都达到了工业级的标准。

### 2. 样式美学 (Styling)

所有的视觉呈现均由 **Tailwind CSS** 构建。这确保了样式的原子性与极高的重构效率。shadcn/ui 并不强制使用某种固定的美学，它提供的是一套极致优雅的“默认样式”，但你可以通过修改 Tailwind 的配置文件或直接在组件源码中重写类名，瞬间将其转化为任何你想要的风格——从极简冷淡到华丽繁复。

### 3. 组件所有权 (Ownership)

本项目存在的唯一目的是赋予开发者以“权力”。当你需要修改一个 Button 在特定业务下的加载态表现，或者需要重构 Dialog 的层级顺序时，你不需要去通过复杂的 API 或 CSS 注入来“欺骗”第三方库，你只需要打开本地文件，根据业务需求进行重写。

## 为什么选择 shadcn/ui？

- **零依赖运行时**：因为它不是一个包，所以你的应用不会因为引入组件而产生额外的运行时依赖开销。
- **无障碍支持**：原生遵循 WAI-ARIA 规范，确保你的应用不仅看起来漂亮，而且对所有障碍群体都具备可操作性。
- **极致的开发体验**：代码直接暴露在你的编辑器中，类型定义清晰，修改即时见效。这种直观性让它在 AI 时代更具优势，各种 LLM 可以直接基于源码为你提供优化建议。
- **构建属于你的设计系统**：它是构建公司内部私有组件库的最佳底座。你可以基于 shadcn/ui 的高抽象标准，快速封装出完全符合企业视觉规范的系统级组件。

---

shadcn/ui 改变了我们思考 UI 基础设施的方式：它不再是外挂的插件，而是你应用基因的一部分。

## 🚀 项目开发与自动部署指南

本项目基于最新的 **Next.js 16** 和 **React 19** 进行了重构，并为 GitHub Pages 配置了完整的 CI/CD 自动化部署。

### 💻 本地开发

1. 安装依赖：
   ```bash
   npm install
   ```

2. 运行本地开发服务器：
   ```bash
   npm run dev
   ```
   在浏览器中打开 [http://localhost:3000](http://localhost:3000) 即可预览。

3. 生产环境静态打包测试：
   ```bash
   npm run build
   ```
   打包完成后，静态页面将输出至 `out/` 文件夹。

### 🌐 自动部署 (GitHub Pages)

项目已集成 GitHub Actions。每次向 `main` 分支提交或推送代码时，部署流水线会自动触发：
1. 流水线会自动拉取代码、构建 Next.js 静态站点。
2. 将构建后的静态页面自动推送到 `gh-pages` 分支以完成部署。
3. 项目中包含 `public/CNAME` 配置文件，部署后会自动将你的自定义域名 **`shadcn.ndjp.net`** 绑定到 GitHub Pages。
